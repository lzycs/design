摘 要

随着智慧校园建设不断推进，高校学习空间管理从传统静态管理逐步转向动态化、精细化与服务化。针对现有校园学习空间系统中普遍存在的功能分散、信息更新不及时、预约与使用环节脱节、反馈处理闭环不完整等问题，本文设计并实现了一套基于实时状态的校园学习空间综合服务平台。系统采用前后端分离架构，前端基于 Vue 3、Pinia 与 Vue Router 构建多角色交互界面，后端基于 Spring Boot、MyBatis-Plus、MySQL 与 Redis 实现业务逻辑与数据持久化，并结合 WebSocket 支持状态实时推送。

本系统围绕校园学习空间使用全流程，构建了“查询-预约-签到-使用-反馈-治理”的业务闭环。用户端实现了教学楼与教室浏览、课程占用信息查看、学习空间预约、二维码签到核销、教室评价、设施报修、协作组队、学习计划管理、资源市场及校园导航等功能；管理端实现了报修工单管理、评价审核、资源审核、教室与课程管理、教学楼管理、预约规则配置以及扫码设备管理等功能。通过统一接口规范、角色权限控制和模块化设计，平台在功能完整性与可维护性方面具备较好工程实践价值。

本文在需求分析基础上给出了系统总体架构、功能模块划分与数据库设计思路，并结合关键业务流程对主要实现方案进行说明。系统测试结果表明，平台能够稳定支撑核心业务场景，满足学习空间综合服务平台在功能可用性、流程完整性与管理协同性方面的设计目标。研究结果可为高校学习空间数字化治理提供可落地的系统实现参考。

关键词：学习空间；实时状态；预约签到；智慧校园；协作服务


Abstract

With the continuous development of smart campus initiatives, learning space management in universities is shifting from static management to dynamic, refined, and service-oriented operations. To address common problems in existing campus systems, such as fragmented functions, untimely information updates, disconnected reservation and usage processes, and incomplete feedback loops, this thesis designs and implements a real-time-status-based integrated service platform for campus learning spaces.

The system adopts a front-end and back-end separated architecture. The front end is built with Vue 3, Pinia, and Vue Router to provide interactive interfaces for multiple roles, while the back end is developed with Spring Boot, MyBatis-Plus, MySQL, and Redis for business logic processing and data persistence. WebSocket is introduced to support real-time status updates.

Centered on the full lifecycle of learning space usage, the platform establishes a closed-loop workflow of "query-reservation-check-in-usage-feedback-governance". The user side supports building and classroom browsing, course occupancy viewing, learning space reservation, QR-code-based check-in and verification, classroom reviews, facility repair reporting, collaboration team functions, study plan management, resource marketplace, and campus navigation. The admin side supports repair ticket management, review moderation, marketplace audit, classroom and course management, building management, reservation rule configuration, and scan-device management.

Based on requirement analysis, this thesis presents the overall architecture, functional module decomposition, and database design strategy, and explains major implementation approaches through key business workflows. Test results show that the system can stably support core scenarios and meet the design goals in terms of functional availability, process integrity, and management collaboration. The research provides a practical reference for the digital governance of campus learning spaces.

Keywords: learning space; real-time status; reservation and check-in; smart campus; collaborative service

第1章 绪论

1.1 研究背景

1.1.1 智慧校园建设推动学习空间管理升级

近年来，教育数字化转型持续推进，智慧校园建设已成为高校信息化建设的重要方向。与教学、科研、管理等业务系统相比，学习空间管理长期存在“系统支持弱、业务协同低、用户体验差”的共性问题。学生在实际学习过程中，不仅需要知道“是否有空教室”，还关心环境舒适度、设施可用性、空间位置、预约规则及协作学习支持等信息。传统以静态公告和人工管理为主的模式，已难以满足高校高频、多样化、实时化的学习空间服务需求。

随着移动互联网、物联网和实时通信技术的普及，高校具备了将学习空间服务从“信息发布”升级为“综合治理与协同服务”的技术条件。通过整合空间状态、课程占用、预约记录、用户反馈和管理流程，能够实现学习空间服务的全流程数字化，进而提升资源利用效率和师生使用体验。

1.1.2 现有校园学习空间系统存在的问题

当前多数高校相关系统仍以“空教室查询”或“单点预约”功能为主，存在以下不足：

（1）功能割裂明显。查询、预约、签到、反馈、报修等业务彼此独立，用户在不同流程之间切换成本高，无法形成完整使用链路。  
（2）实时性不足。空间状态更新机制不完善，用户看到的信息与现场状态可能不一致，影响预约决策与到场体验。  
（3）治理闭环不完整。用户评价与报修信息缺乏有效流转机制，管理端难以形成可追踪、可分析、可改进的治理路径。  
（4）协作支持不足。对小组学习、共享计划、团队沟通等协作场景支持有限，难以满足复合型学习需求。

以上问题表明，学习空间管理不应仅停留在“查询工具”层面，而应向“综合服务平台”演进。

1.1.3 课题研究的必要性

基于实时状态的校园学习空间综合服务平台能够有效连接学生端与管理端，形成以用户需求为导向、以数据驱动为支撑的服务体系。一方面，平台可通过预约规则约束、签到核销、实时反馈机制提升空间流转效率，减少资源闲置与重复占用；另一方面，平台可将评价、报修、审核、配置等管理行为纳入统一系统，实现服务质量持续优化。由此，本课题具有明确的实践价值与推广意义。

1.2 国内外研究现状

1.2.1 国外研究现状

国外高校在学习空间数字化管理方面起步较早，相关平台通常具备较强的系统整合能力与数据驱动能力。部分高校已将教室占用、图书馆座位管理、空间导航、环境监测等功能统一到校园移动平台中，并通过可视化统计支持资源优化配置。研究重点逐步由“是否可用”扩展到“如何高效使用”和“如何提升服务体验”，在实时状态采集、多源数据融合、行为分析与个性化推荐方面形成了较成熟思路。

1.2.2 国内研究现状

国内高校近年来在智慧教室、空教室查询、图书馆预约等领域发展迅速，系统覆盖率不断提升。但从整体上看，仍存在功能模块分散、系统联动不足、治理闭环不完整等问题。多数应用围绕“查询与预约”展开，对签到核销、评价反馈、报修流转、协作学习等后续环节支持较弱；同时，管理侧对用户行为数据的利用深度有限，难以形成持续改进机制。

1.2.3 研究评述

综合国内外现状可见，学习空间服务平台建设正在从“单功能应用”向“全流程协同平台”转变。现有研究与实践为本课题提供了技术基础，但在业务闭环构建、角色协同治理、模块化工程实现等方面仍有提升空间。本文以真实校园场景为导向，重点解决查询、预约、签到、反馈、治理的衔接问题，强调系统可实现性与落地性。

第3章 需求分析与总体设计

3.1 业务需求分析

3.1.1 角色划分

系统面向多角色使用场景，主要角色包括：

（1）学生用户：核心使用者，负责空间查询、预约签到、评价报修、协作组队、学习计划与资源交易等操作。  
（2）管理员：负责教室、课程、教学楼等基础数据维护，以及预约规则配置与系统治理。  
（3）审核/运维角色：包括评价审核人员、资源审核人员、维修处理人员等，负责内容审核与工单流转处理。

不同角色通过权限边界区分功能入口，实现“前台服务 + 后台治理”的协同机制。

3.1.2 核心业务流程

结合项目实现情况，系统核心业务流程可归纳为：

（1）空间使用流程：查询空间 -> 查看时段占用 -> 发起预约 -> 生成预约二维码 -> 到场扫码核销 -> 使用完成。  
（2）反馈治理流程：用户提交评价/报修 -> 管理端审核与处理 -> 结果反馈与状态更新。  
（3）协作学习流程：发起小组/提交申请 -> 组长审核 -> 进入小组沟通 -> 关联学习计划与研讨室预约。  
（4）运营管理流程：管理员维护基础资源 -> 配置预约规则 -> 处理审核与工单 -> 监控系统运行状态。

3.2 功能需求分析

3.2.1 用户端功能需求

（1）首页与信息浏览  
支持教学楼列表、教室列表、课表与空间详情查看，帮助用户快速定位学习资源。

（2）学习空间预约与签到  
支持教室/座位预约，支持预约规则约束（每周上限、单次时长上限），支持二维码生成、扫码核销与预约状态查询。

（3）评价与报修  
支持提交教室评价、查看个人评价记录、查看公开评价；支持报修提交与进度查询，形成使用后反馈链路。

（4）协作组队与沟通  
支持创建小组、加入/退出小组、入组申请审核、消息中心提醒、小组聊天等协作能力。

（5）学习计划  
支持创建和维护共享学习计划，并与预约记录关联，强化团队学习组织性。

（6）资源市场与个人中心  
支持资源发布、交易意向、交易完成确认以及个人资源数据管理，扩展平台服务边界。

（7）校园导航  
支持校园点位搜索、目标选择、路线规划与预约场景联动导航。

3.2.2 管理端功能需求

（1）认证与权限菜单：支持管理员登录与按权限返回菜单。  
（2）报修工单管理：支持工单查看、状态更新与处理记录。  
（3）评价与资源审核：支持评价审核、资源市场发布审核。  
（4）基础数据管理：支持教室、课程、教学楼维护。  
（5）规则与设备管理：支持预约限制配置、扫码设备授权与启停管理。

3.3 非功能需求分析

3.3.1 可用性需求

系统需支持清晰的页面层级与稳定的交互流程，保证用户在移动端与桌面端均可快速完成操作。对于高频场景（如预约、签到、报修），应减少步骤复杂度并提供明确结果反馈。

3.3.2 实时性需求

学习空间服务对实时性要求较高。系统需能够及时反馈预约状态、签到核销结果及部分消息通知，降低信息滞后导致的使用冲突与体验损耗。

3.3.3 性能需求

在选课周、考试周等高峰时段，预约请求会集中出现。系统需通过缓存与合理的数据访问策略保障核心接口响应效率，并保证业务规则校验的正确性。

3.3.4 安全性需求

系统需实现角色权限隔离、关键接口鉴权、输入参数合法性校验，并保障用户隐私数据存储与传输安全。管理端操作应具备可追踪性，避免越权与误操作风险。

3.3.5 可维护性需求

系统需具备良好的模块边界与扩展能力。前后端分离、接口统一返回结构、数据层规范化设计能够降低后续功能迭代成本。

3.4 系统总体设计

3.4.1 总体架构设计

系统采用前后端分离架构。前端负责页面展示、交互控制与接口调用；后端负责业务编排、规则校验、数据读写与消息推送；MySQL 提供结构化数据存储，Redis 提供缓存能力，WebSocket 提供实时通信支持。该架构能够满足本课题在功能复杂度、实时性与可扩展性方面的需求。

3.4.2 前端架构设计

前端以 Vue 3 为核心，使用 Vue Router 管理路由，Pinia 管理跨页面状态，Axios 封装接口请求。视图层按业务模块组织，如预约、反馈、协作、导航、个人中心与管理端入口。该设计有利于页面解耦与功能并行开发。

3.4.3 后端架构设计

后端基于 Spring Boot 构建，采用 Controller-Service-Mapper 分层模式。Controller 层负责参数接收与接口响应，Service 层负责业务逻辑与流程控制，Mapper 层负责数据库访问。通过统一响应结构与异常处理机制，保证接口风格一致性和维护便利性。

3.4.4 业务闭环设计

本系统核心设计思想是构建学习空间服务闭环：  
查询（空间信息） -> 预约（规则约束） -> 签到（二维码核销） -> 使用（学习活动） -> 反馈（评价/报修） -> 治理（审核/处理/配置优化）。  
该闭环连接用户行为与管理行为，使系统不仅“可用”，也“可管、可优化”。

3.5 数据库总体设计

3.5.1 实体识别

根据业务需求识别核心实体：用户、教学楼、教室、课程、预约记录、签到核销、评价记录、报修工单、协作小组、入组申请、小组消息、学习计划、资源条目、交易记录、扫码设备等。

3.5.2 关系设计原则

数据库设计遵循以下原则：  
（1）以业务流程为主线建立主外键关联，保障数据可追溯；  
（2）以角色权限为约束划分管理字段，保障数据治理需求；  
（3）以高频查询场景为导向设计索引策略，提升接口响应效率；  
（4）以逻辑删除策略维护历史记录，兼顾审计与恢复需求。

3.5.3 关键数据流说明

预约数据与教室、用户、时间槽关联；评价与报修数据与用户及空间实体关联；协作小组与成员、消息、学习计划关联；管理端通过审核与工单状态字段实现流程推进。通过上述设计，可支持系统多模块联动运行。

3.6 本章小结

本章围绕系统建设目标完成了需求分析与总体设计，明确了角色职责、功能边界、非功能要求、架构方案和数据库设计思路，为后续详细实现与测试验证提供了结构化依据。

第2章 相关技术与理论基础

本章围绕系统实现所涉及的前端框架、后端框架、数据存储与中间件、实时通信与二维码核销等关键技术进行阐述，并从“技术原理—选型理由—本系统落地方式”的角度说明其适用性与实现要点，为第4章的系统实现提供理论与技术支撑。

2.1 前端关键技术

2.1.1 Vue 3 组件化开发模式

（1）技术原理  
Vue 采用“组件（Component）”作为最小复用单元，将界面拆解为可组合的结构。组件内部包含模板（Template）、逻辑（Script）与样式（Style），通过 Props、事件（emit）与插槽（slot）实现组件间数据与结构的传递。在复杂业务页面中，组件化有助于将“展示”和“交互逻辑”以模块边界隔离，从而降低单文件复杂度。

（2）选型理由  
学习空间综合服务平台页面多、业务流程长、角色多样（用户端 + 管理端）。Vue 3 具备成熟生态与高开发效率，适合快速迭代和模块化扩展；配合按需加载与路由拆分，可有效控制前端包体与首屏加载时间。

（3）本系统落地方式  
本系统在用户端采用“预约中心—教室详情—二维码核销—反馈/报修—协作—计划—资源市场—导航”等多页面结构，天然适配组件化开发；在管理端采用统一管理框架页面承载多管理入口，便于后续扩展治理功能。

2.1.2 Vue 3 响应式系统与数据驱动视图

（1）技术原理  
Vue 3 的响应式系统通过依赖追踪实现“状态变化自动触发视图更新”。当响应式状态被读取时，会收集依赖；当状态被修改时，会触发相关依赖的重新计算与渲染。该机制能够减少手工 DOM 更新，并将界面一致性建立在状态一致性之上。

（2）在本系统中的典型场景  
预约状态变化（待签到/已核销）、协作红点角标变化、评价列表刷新、报修工单状态变化等均可通过状态更新驱动界面变化，从而提升交互连贯性与可维护性。

2.1.3 Pinia 状态管理

（1）技术原理  
在单页应用中，跨页面共享数据若仅依靠组件层层传递会导致“数据来源不清、维护成本高”。Pinia 以 Store 为核心，提供 state、getter、action 三类能力：state 保存共享状态，getter 派生计算数据，action 封装异步请求与业务操作，形成集中可控的数据流。

（2）选型理由  
相比传统 Vuex，Pinia 更轻量、API 更贴近组合式 API，学习成本低、工程组织更清晰。对于本系统这类多模块应用，Pinia 能降低跨页面状态同步难度，提升开发效率。

（3）本系统落地方式  
本系统存在登录态与用户信息、协作消息提醒、预约跳转携带目标地点等跨页面共享需求，适合使用 Pinia 统一管理，避免“页面间隐式耦合”。

2.1.4 Vue Router 与前端路由治理

（1）技术原理  
Vue Router 通过 History API 实现 URL 与页面组件映射，使应用在不刷新页面的情况下完成视图切换。路由不仅负责导航，还可承载路由参数（如 buildingId、roomId）、页面级懒加载以及路由守卫（鉴权控制）等能力。

（2）本系统落地方式  
系统路由覆盖用户端与管理端入口：用户端包含首页、预约、反馈、协作、导航与个人中心等页面；管理端包含后台登录与管理页面集合。通过页面懒加载可降低首屏资源加载压力，提升移动端体验。

2.1.5 Axios 与统一接口封装

（1）技术原理  
Axios 提供统一的 HTTP 请求能力，并支持请求/响应拦截器机制。拦截器可用于注入鉴权信息（如 token）、统一错误处理（如网络异常、接口返回异常码）以及统一解析返回结构。

（2）本系统落地方式  
前端对后端接口采用统一封装：  
1）统一配置 API Base URL，保证多环境切换便利；  
2）通过拦截器集中处理登录过期、权限不足等异常提示；  
3）对管理端请求可按后端约定注入管理端 token（例如通过请求头传递），从而实现用户端与管理端鉴权策略隔离。

2.1.6 ECharts 可视化（扩展支撑）

学习空间平台具备天然的统计分析价值，如空间使用热度、热门时段、报修类型分布等。ECharts 提供丰富图表与交互能力，可用于管理端看板与运营分析。本系统引入 ECharts 作为可视化能力储备，便于后续将“数据治理”从功能层扩展到分析层。

2.2 后端关键技术

2.2.1 Spring Boot 与分层架构（Controller-Service-Mapper）

（1）技术原理  
Spring Boot 提供自动配置、依赖管理与约定优于配置等能力，适合快速构建可运行的 Web 服务。分层架构中：  
Controller 层负责参数接收、鉴权入口与返回结果封装；  
Service 层负责业务流程编排与规则校验；  
Mapper 层负责数据库访问与数据对象映射。  
分层架构能显著降低“接口逻辑与业务逻辑、业务逻辑与数据访问”之间的耦合。

（2）本系统落地方式  
本系统以 REST 接口形式对外提供服务，后端统一返回结构（便于前端统一处理），并将预约规则校验、二维码核销、审核流转等核心逻辑封装在 Service 层，提升复用性与可测试性。

2.2.2 MyBatis-Plus 数据访问框架

（1）技术原理  
MyBatis-Plus 在 MyBatis 的基础上提供通用 CRUD、条件构造器、分页等能力，减少重复 SQL 编写。其实体映射机制能够将表结构与实体对象对应，配合全局配置可实现逻辑删除字段的统一管理。

（2）选型理由与落地方式  
本系统数据表数量较多且增删改查场景丰富（预约、评价、报修、协作、资源市场等），使用 MyBatis-Plus 能显著提升开发效率，并统一数据层编码风格。

2.2.3 MySQL 事务与一致性保障

（1）技术原理  
MySQL 支持 ACID 事务，可用于保证多步骤更新的一致性，例如“创建预约记录—写入时间槽占用—生成核销状态”等场景。通过合理的索引设计，可在保证一致性的同时提升查询性能。

（2）本系统落地方式  
系统核心数据（用户、空间资源、预约、评价、报修、协作、交易等）均采用 MySQL 存储。预约、核销、审核与工单流转等关键流程需要保证状态更新的一致性与可追溯性，适合使用关系型数据库承载。

2.2.4 Redis 缓存与高频读优化

（1）技术原理  
Redis 基于内存存储，适合存放访问频率高、更新频率相对可控的数据（热数据），能够显著降低数据库压力。常见使用方式包括：  
1）缓存查询结果（如空间列表、预约余量、二维码状态短期缓存）；  
2）存放临时状态（如短期 token、验证码、设备配对状态等）；  
3）作为分布式协调组件（后续可扩展分布式锁、防超订等）。

（2）本系统落地方式  
系统在预约与二维码状态查询等高频场景引入 Redis，提升响应速度与稳定性，并为后续的高并发控制策略预留扩展空间。

2.2.5 WebSocket 实时通信机制

（1）技术原理  
WebSocket 在浏览器与服务端之间建立长连接，支持双向消息推送。与轮询相比，WebSocket 能减少无效请求并降低状态变更传递延迟，适合实时通知与状态更新。

（2）本系统落地方式  
系统在“预约状态变化提示”“协作消息提醒”“管理端处理结果通知”等场景中具备实时推送需求，引入 WebSocket 能提升用户端体验与管理协同效率。

2.2.6 Spring Boot 文件上传与多媒体数据处理（用于报修图片）

（1）技术原理  
Spring Boot 通过 Multipart 机制支持文件上传，可在服务端进行文件大小限制、类型校验与存储路径管理。对报修图片等多媒体数据，通常采用“文件存储 + 数据库保存元信息”的方式：数据库保存文件 URL/路径、关联工单 ID、上传时间等，文件本体存放于本地或对象存储。

（2）本系统落地方式  
系统支持报修图片上传，并在后端配置文件上传大小限制，保证在常见网络环境下上传可靠且不影响服务稳定性。

2.3 关键业务技术原理

2.3.1 “实时状态”的业务定义与数据融合思路

学习空间的“实时状态”不是单一数据源，而是多源信息融合后的对外展示结果。典型数据源包括：  
（1）课程排课：决定教室在某时间段是否被教学占用；  
（2）预约记录：决定教室/座位在某时间段是否被用户预约；  
（3）核销状态：反映预约是否已到场确认，可用于辅助判断“占而不到”等问题；  
（4）报修状态：若空间设施报修中，可提示用户选择其他空间或给出风险提醒。  

在工程实现上，可将状态展示抽象为“按时间段的占用判定”，由后端统一计算后下发前端，避免前端分别调用多接口并在页面层拼装导致的不一致风险。

2.3.2 预约规则约束与冲突校验

预约模块需要满足“规则可配置、校验可复用、结果可解释”。常见约束包括：  
（1）次数约束：每周预约次数上限；  
（2）时长约束：单次预约最长时长与连续时段上限；  
（3）冲突约束：课程占用冲突、已预约冲突、时间槽不可用等；  
（4）状态约束：已核销/已取消/已过期等状态不可重复操作。  

系统应在服务端集中完成校验并返回明确错误信息，前端将错误信息以用户可理解方式提示，从而提升可用性与系统可信度。

2.3.3 二维码生成与扫码核销

（1）技术原理  
二维码本质是对一段信息的编码，扫码核销需要解决两个核心问题：  
1）二维码内容如何与预约记录唯一绑定；  
2）扫码提交后如何进行有效性校验与状态更新。  

（2）典型校验点  
校验预约是否存在、是否属于有效时间窗口、是否已核销、是否被取消/过期等，并在核销成功后更新预约状态，返回可追溯结果，防止重复核销或伪造二维码。

2.3.4 评价审核与内容治理

公开评价既能为后续用户提供参考，也存在不当内容风险，因此需建立审核机制。典型策略为：用户提交评价后进入待审核状态；审核通过后对外公开；审核拒绝则不公开但保留记录（便于申诉与追溯）。该机制体现了平台从“工具”走向“治理”的能力建设。

2.3.5 报修工单机制与状态机

报修业务可抽象为有限状态机：待处理 -> 处理中 -> 已完成（也可扩展已驳回/已关闭）。通过状态机可将“用户发现问题—管理响应—处理完成”形成闭环，并沉淀可统计数据（故障类型、处理时长、集中故障点位等），为后续优化提供依据。

2.3.6 RBAC 权限控制（用户端与管理端隔离）

系统存在多角色、多入口的权限需求。RBAC（Role-Based Access Control）通过“用户—角色—权限”关系实现访问控制：用户被分配角色，角色拥有权限集合，接口或菜单依据权限开放。  
本系统将用户端与管理端进行隔离：用户端围绕学习服务；管理端围绕审核、工单与配置等治理能力。通过鉴权头与菜单权限控制，可降低越权风险并提升后台操作安全性。

2.4 本章小结

本章从前端、后端与关键业务机制三个层面系统介绍了平台实现所需的技术基础，并阐述了这些技术在学习空间综合服务场景中的适用性。通过前后端分离、分层架构、缓存与实时通信等手段，系统能够在保证开发效率的同时满足实时性、稳定性与可扩展性要求，为后续系统实现与测试分析提供技术支撑。

第4章 系统详细设计与实现

4.1 系统总体实现说明

系统实现以业务闭环为主线，围绕学习空间服务流程实现各模块联动。用户端关注“效率与体验”，管理端关注“治理与可运营”。系统通过统一接口规范和模块化设计，将复杂业务拆解为相对独立的子模块，包括学习空间与课表、预约与核销、评价与报修、协作与消息、学习计划、资源市场、校园导航以及后台管理模块。

4.2 核心业务流程设计与实现

4.2.1 学习空间预约业务流程

预约业务流程可概括为：用户选择空间与日期 -> 查询可用时段 -> 选择连续时段 -> 提交预约 -> 返回预约记录与二维码 -> 在“我的预约”中管理状态。

为保证可用性与一致性，系统在服务端完成关键校验：  
（1）规则校验：每周预约次数上限、单次时长上限；  
（2）冲突校验：课程占用、已预约时间段冲突；  
（3）可解释错误：返回明确失败原因，便于用户调整选择。

图4.1 学习空间预约流程图（建议绘制）
节点建议：开始 -> 选择空间/日期 -> 查询时段占用 -> 选择时段 -> 服务端校验（规则/冲突） -> 预约成功（生成二维码）/预约失败（提示原因） -> 结束。

4.2.2 二维码生成与扫码核销流程

二维码核销用于确认用户到场，提高空间资源周转率。系统在预约成功后生成二维码内容并与预约记录绑定；扫码端在核销时向后端提交二维码信息，后端校验二维码有效性、预约状态与时间窗口，并更新预约状态。

图4.2 二维码核销流程图（建议绘制）
节点建议：预约生成二维码 -> 用户到场 -> 扫码端提交二维码 -> 服务端校验（是否存在/是否过期/是否已核销） -> 更新状态 -> 返回核销结果 -> 前端刷新预约状态。

4.2.3 评价审核与公开展示流程

评价模块遵循“个人可管理、公开需审核”的治理思路。用户提交评价后进入待审核状态；审核人员在管理端进行审核，通过后才对外展示在教室详情页与“全部评价”页。该流程既保证用户反馈渠道畅通，也保证公共内容质量。

图4.3 评价数据流与审核流程图（建议绘制）
节点建议：提交评价 -> 存储评价记录（待审核） -> 管理端审核（通过/拒绝） -> 通过：进入公开列表；拒绝：不公开但保留记录 -> 用户可查看个人评价记录与删除。

4.2.4 报修工单流转流程

报修模块用于设施治理与服务提升。用户选择空间并提交图文报修信息，系统生成工单；维修/管理人员在后台查看工单并推进状态，如“待处理-处理中-已完成”。通过工单流转，平台能够沉淀设备问题数据，支持后续统计分析与维护决策。

图4.4 报修工单流转流程图（建议绘制）
节点建议：提交报修 -> 生成工单（待处理） -> 后台接单 -> 处理中 -> 已完成 -> 用户查看进度。

4.2.5 协作组队与消息通知流程

协作模块支持用户创建学习小组与加入申请，组长进行审核；通过后成为成员并进入小组功能。系统提供消息中心用于聚合协作相关通知，同时支持小组聊天，增强协作沟通效率。

图4.5 协作组队与审核流程图（建议绘制）
节点建议：创建小组/浏览小组 -> 提交加入申请 -> 组长审核（通过/拒绝） -> 结果通知 -> 进入小组（成员列表/聊天/计划）。

4.2.6 学习计划与预约联动流程

学习计划用于将“目标与时间安排”结构化，并与空间预约关联形成执行闭环。用户可按小组或按预约关联查询学习计划，便于团队在固定时段与固定空间开展协作学习。

图4.6 学习计划与预约联动流程图（建议绘制）
节点建议：创建/更新学习计划 -> 选择关联小组/关联预约 -> 保存计划 -> 在共享计划页展示 -> 从计划跳转到相关空间/预约信息。

4.3 功能模块实现要点（按用户端/管理端）

4.3.1 用户端模块实现要点

（1）空间浏览与课表：支持教学楼、教室与课表信息查询，便于用户选择空间。  
（2）预约中心：支持按日期切换与资源分类浏览；教室详情页展示时段占用并支持连续时段预约。  
（3）我的预约：支持查看预约记录、待签到状态与一键跳转导航。  
（4）反馈中心：包含待评价、提交评价、个人评价管理、公开评价查看、报修提交与记录查询。  
（5）协作与计划：支持小组管理、消息中心、小组聊天与共享学习计划。  
（6）资源市场：支持发布、浏览、交易意向、交易完成确认与个人资源管理。  
（7）校园导航：支持地图浏览、点位搜索、定位与路线规划（可用时为路径服务，不可用时降级直线导航）。

4.3.2 管理端模块实现要点

（1）后台登录与菜单权限：根据权限返回后台菜单，避免越权入口。  
（2）报修工单管理：支持工单列表、状态更新与处理过程记录。  
（3）评价审核：支持审核通过/拒绝，控制评价公开展示。  
（4）资源审核：对资源市场内容进行审核，保障内容合规。  
（5）基础数据管理：支持教室、课程、教学楼等数据维护。  
（6）预约限制配置：可配置每周预约上限、单次时长等规则。  
（7）扫码设备管理：支持生成配对 token、设备授权绑定、启停与删除。

4.4 数据库与核心表设计（示例描述）

为支撑平台业务闭环，数据库至少应覆盖以下核心数据域：
（1）用户域：用户基本信息、角色与状态；  
（2）空间域：教学楼、教室、图书馆座位等资源信息；  
（3）预约域：预约记录、时间槽、核销状态、二维码状态；  
（4）反馈域：评价记录、审核状态、报修工单与处理状态；  
（5）协作域：小组、成员、申请、消息、聊天记录；  
（6）计划域：学习计划与关键节点；  
（7）市场域：资源条目、交易意向与交易记录；

第6章 总结与展望

6.1 工作总结

本文围绕“基于实时状态的校园学习空间综合服务平台”开展研究与实现工作，面向高校学习空间管理中功能分散、流程割裂与治理闭环不足等问题，设计并实现了覆盖用户端与管理端的一体化平台。系统以“查询-预约-签到-使用-反馈-治理”为主线，完成了学习空间预约、二维码核销、评价审核、报修工单流转、协作组队、学习计划联动、资源市场与校园导航等功能，并提供后台管理能力支撑内容审核与运营治理。

在工程实现方面，系统采用前后端分离架构与模块化设计，后端通过统一接口规范与分层结构提高可维护性，结合缓存与实时通信机制提升用户体验与系统实时性。测试章节对核心流程与异常场景进行了验证，证明系统具备落地可用性与一定的推广价值。

6.2 不足与改进方向

尽管系统已实现主要业务闭环，但仍存在以下不足：
（1）实时状态数据精度受限：部分空间状态依赖课程与预约推断，缺少更丰富的真实感知数据来源；  
（2）智能化能力不足：对空间推荐、热度预测与拥挤程度预测等智能决策支持有待加强；  
（3）运营与统计能力仍可增强：管理端可进一步完善数据看板、告警与运维监控能力；  
（4）部署与安全体系需深化：可进一步完善生产级部署、权限细化、审计日志与安全防护策略。

6.3 未来展望

未来可从以下方向持续完善：
（1）引入更丰富的状态感知与融合策略，例如与门禁、Wi-Fi 连接数、传感器数据等结合，提高实时性与准确性；  
（2）增加个性化推荐与预测模块，基于历史使用行为与热门时段统计，为用户提供更贴合需求的空间推荐；  
（3）完善消息推送与提醒体系，实现预约提醒、违约提醒、报修进度推送等；  
（4）扩展多校区/多场馆模型与跨端部署能力，提升系统在更大规模场景下的可迁移性与可运营性。

6.4 本章小结

本文完成了校园学习空间综合服务平台的设计与实现，构建了以实时状态为基础的服务闭环，并通过测试验证了系统可用性。未来将继续围绕数据驱动治理与智能化服务能力进行拓展与优化。
