第3章 需求分析与总体设计

本章在任务书与开题报告的目标约束下，结合系统当前已实现功能，完成角色与业务场景梳理、功能与非功能需求归纳，并给出系统总体架构、模块划分与数据库总体设计，为第4章的详细设计与实现提供依据。

3.1 业务需求分析

3.1.1 角色划分与职责

系统面向校园学习空间全流程服务，按照“前台服务 + 后台治理”的思路划分多角色协同：

（1）学生用户（普通用户）  
主要任务：浏览教学楼/教室信息、查询占用与可用时段、发起预约并完成签到核销、提交教室评价与报修、参与协作小组与聊天、维护学习计划、参与资源市场交易、使用校园导航到达目标点位等。

（2）教师用户（扩展角色，可与学生同属前台用户体系）  
主要任务：除学生能力外，更偏向于协作学习组织与固定空间使用需求（实际功能以系统实现为准）。在论文描述中可作为“前台用户的一类角色”，用于说明平台支持多类型用户。

（3）管理员（后台治理角色）  
主要任务：后台登录与菜单访问、教学楼/教室/课程基础数据维护、预约规则（周上限、单次时长等）配置、扫码设备管理、平台概览统计（若已实现）等。

（4）评价审核员（内容治理角色）  
主要任务：审核用户评价内容，控制公开展示范围，屏蔽不当言论，保证公共评价可参考性。

（5）资源市场审核员（内容治理角色）  
主要任务：对资源市场发布内容进行审核（通过/拒绝），保障平台内容合规。

（6）维修/运维人员（工单处理角色）  
主要任务：查看报修工单，推进工单状态（待处理/处理中/已完成），形成设备治理闭环。

3.1.2 典型业务场景与用户目标

为确保平台功能贴近校园真实使用，本研究将业务归纳为以下典型场景：

（1）临时自习：学生在空闲时间快速查找可用教室/座位并预约，到场签到后使用。  
（2）小组讨论：协作小组需要在固定时间集中学习，要求空间可预约且支持多人协作，并可在计划中关联研讨室预约。  
（3）到场导航：用户预约成功后需要快速到达目标地点，希望系统提供路径规划或直线导航降级方案。  
（4）使用后反馈：用户使用教室后对环境、插座、噪音等进行评价；若发现设备故障则提交报修并跟踪处理进度。  
（5）运营治理：管理员/审核/维修人员在后台对评价、报修、资源发布等进行治理，保障平台秩序并提升服务质量。

3.1.3 核心业务流程（细化）

（1）学习空间使用闭环  
查询空间信息 → 查询时段占用 → 选择连续时段 → 提交预约（规则/冲突校验） → 生成二维码 → 到场扫码核销 → 使用完成 → 评价/报修 → 后台治理。

（2）内容治理闭环  
用户提交评价/资源 → 后台审核 → 通过后公开展示/交易可见；拒绝则不公开但可留痕追溯。

（3）设施治理闭环  
用户提交报修（图文） → 后台接单 → 状态推进 → 完成闭环 → 数据沉淀（类型、频次、处理时长）。

图3.1 学习空间服务闭环流程图（建议绘制）
绘制要点：将“预约-核销-反馈-治理”放入同一闭环，并标注关键节点的责任角色（用户端/管理端）。

3.2 功能需求分析

3.2.1 用户端功能需求（按模块细化）

（1）首页与基础浏览模块  
功能点：  
1）提供平台主要服务入口；  
2）教学楼列表与详情展示；  
3）按教学楼查看教室列表与教室详情；  
4）楼宇导航与导航详情展示；  
5）教学楼课表/排课信息展示。  
需求目标：降低用户信息获取成本，支持“先查再约”的决策路径。

（2）学习空间预约模块（教室/图书馆座位）  
功能点：  
1）预约中心按日期切换，支持资源分类（教室/座位等）；  
2）教室详情展示时段占用情况与可预约时间段；  
3）支持连续时段预约；  
4）预约规则读取并限制：每周预约次数上限、单次预约最长时长；  
5）个人预约记录查询、状态展示与跳转；  
6）预约二维码生成与展示；  
7）预约签到入口与核销状态查询；  
8）从“我的预约（待签到）”一键跳转校园导航并带入目标地点。  
需求目标：形成“预约—到场确认—使用”的闭环，提高空间周转效率。

（3）评价与反馈模块  
功能点：  
1）待评价记录查询；  
2）教室评价提交（星级/文字等）；  
3）个人评价记录查看与删除；  
4）教室详情页展示审核通过的评价；  
5）查看全部评价页面；  
6）用户提交报修（支持图片）；  
7）个人报修记录查看。  
需求目标：构建“使用后反馈”链路，提升信息透明度与服务质量。

（4）协作组队与沟通模块  
功能点：  
1）协作广场浏览活跃小组；  
2）查看小组详情与成员列表；  
3）发起协作小组、加入/退出、删除自己创建的小组；  
4）加入申请提交与组长审核（通过/拒绝）；  
5）申请结果查询与待处理申请查询；  
6）协作红点角标提醒；  
7）小组聊天；  
8）消息中心查看协作相关消息；  
9）进入研讨室协作页面。  
需求目标：支持小组学习组织与沟通协作，将“空间”与“人”连接起来。

（5）学习计划模块  
功能点：  
1）共享学习计划页展示；  
2）按协作小组查询学习计划；  
3）按预约查询关联学习计划；  
4）创建“学习计划 + 教室预约”联动记录；  
5）更新学习计划；  
6）计划中关联研讨室、日期、时间段与关键时间节点。  
需求目标：强化协作学习的计划性，将“目标—时间—空间”贯通。

（6）课程表与我的课程模块  
功能点：  
1）按教学楼查看课表（按日/周/月）；  
2）教室课表查询与时段占用状态查询；  
3）按楼层/关键字/时间段筛选日课表；  
4）我的课程按日/周查看；  
5）课程星标、备注、课前提醒时间配置。  
需求目标：将课程占用作为空间状态的重要数据源，避免预约与教学冲突。

（7）资源市场与校园学分模块  
功能点：  
1）资源市场广场浏览与筛选；  
2）发布学习资源/二手资源（标题、描述、类别、价格、图片、推荐交易地点）；  
3）发起交易意向并填写留言与线下交接地点；  
4）交易双方确认完成；  
5）个人发布记录与交易记录；  
6）个人资源管理维护；  
7）查询个人校园学分信息（若已实现）。  
需求目标：扩展平台服务边界，增强用户活跃度与平台价值。

（8）校园导航模块  
功能点：  
1）校园地图浏览（教学楼+公共场馆点位）；  
2）按名称/编号/地址搜索目标点位；  
3）定位“我的位置”并展示与目标距离；  
4）一键规划步行路线（不可用时降级直线导航）；  
5）预约场景携带目标名称自动选中并尝试规划路线。  
需求目标：解决“约到但找不到”的最后一公里问题。

3.2.2 管理端功能需求（按治理流程细化）

（1）后台认证与菜单权限  
支持管理员登录；根据权限返回后台菜单；通过菜单与接口双重限制避免越权。

（2）报修工单管理  
支持工单列表查询、状态更新、处理过程记录与筛选；保障设施治理闭环可追踪。

（3）评价审核  
支持评价列表、通过/拒绝操作；通过后允许在用户端公开展示，拒绝则不公开。

（4）资源市场审核  
支持资源发布内容审核（通过/拒绝）；通过后对外可见并参与交易流程。

（5）基础数据管理  
支持教室管理、课程管理、教学楼管理；支持导入/维护（按系统实现描述）。

（6）预约限制配置  
支持每周预约次数上限、单次最长时长等规则配置；当库表无配置时使用后端默认值兜底。

（7）扫码设备管理  
支持生成设备配对 token、设备端扫码授权绑定 UID、设备列表查看、启用/停用、删除；支撑预约扫码核销场景。

3.3 非功能需求分析

3.3.1 可用性与体验需求

（1）移动端友好：页面布局适配手机端；关键操作尽量在 3 步内完成（例如预约、报修）。  
（2）交互一致性：列表筛选、空态、错误提示、加载状态保持一致。  
（3）可理解性：预约失败必须返回可解释原因（超限/冲突/不可用）。

3.3.2 实时性需求

（1）预约状态与核销结果需要快速反馈（避免用户重复操作）；  
（2）协作消息与红点提醒应及时更新；  
（3）关键状态变化可通过 WebSocket 推送（或轮询兜底，按实现描述）。

3.3.3 性能需求（建议给出可写进论文的指标口径）

（1）常用查询接口：在一般网络条件下应保持较低延迟（论文可用平均响应时间、P95 说明）。  
（2）高峰预约场景：并发预约请求增多时，系统应保证规则校验正确、数据一致，并保持可接受的错误率与响应时间。  
（3）缓存策略：对热点数据（教室列表、占用信息、二维码状态）采用 Redis 缓存降低数据库压力。

3.3.4 安全性需求

（1）鉴权：用户端与管理端 token 隔离；接口需校验登录态与权限。  
（2）输入校验：对关键参数（时间段、资源ID、图片上传等）进行合法性校验。  
（3）数据保护：密码等敏感信息需安全存储（按实现说明），管理端操作具备可追踪性（可扩展审计日志）。

3.3.5 可维护性与可扩展性需求

（1）前后端分离与接口规范：统一返回结构 `Result<T>`，便于前端统一处理；  
（2）模块化：预约、反馈、协作、计划、市场、管理端等模块相对独立；  
（3）配置化：预约规则通过配置表管理，支持无配置兜底默认值。

3.4 系统总体设计

3.4.1 总体架构设计

系统采用前后端分离架构：前端负责页面展示与交互；后端负责业务编排、规则校验、状态计算、数据持久化与消息推送；MySQL 作为主数据存储；Redis 作为缓存与临时状态存储；WebSocket 作为实时消息通道。

图3.3 系统总体架构图（建议绘制）
绘制要点：前端（用户端/管理端）→ REST API → 后端；后端连接 MySQL/Redis；WebSocket 从后端到前端；文件上传链路（报修图片）。

3.4.2 功能模块划分

为保证开发与维护效率，系统按业务域划分模块：
（1）空间与课表模块：教学楼、教室、课表查询、占用状态展示；  
（2）预约与核销模块：预约创建、规则校验、二维码生成、扫码核销、状态查询；  
（3）反馈与治理模块：评价提交/展示/审核、报修提交/工单处理；  
（4）协作与消息模块：组队、申请审核、消息中心、聊天、协作页面；  
（5）学习计划模块：计划创建/更新、与小组/预约联动；  
（6）资源市场模块：资源发布/审核/交易；  
（7）校园导航模块：点位搜索、定位与路线规划；  
（8）后台管理模块：基础数据、规则配置、扫码设备管理等。

3.4.3 数据流与接口风格设计

（1）统一返回结构：后端对外统一返回 `Result<T>`，包含 code、message 与 data，便于前端统一处理成功/失败。  
（2）接口分域：按业务域组织接口（如预约、评价、报修、协作等），降低耦合。  
（3）状态驱动：重要状态在后端集中计算与校验（例如预约冲突与超限），前端以展示与交互为主，避免前端自行拼装造成不一致。

3.4.4 关键状态模型（建议在论文中用表格表达）

表3.2 预约状态模型表
状态值 | 状态名称 | 业务含义 | 允许操作 | 典型触发条件
1 | 待签到 | 预约已创建，等待到场核销 | 生成二维码、扫码核销、取消预约 | 用户提交预约成功

3.5 数据库总体设计

3.5.1 主要实体识别

结合系统功能模块，数据库核心实体包括：
用户（User）、角色/权限（Role/Permission，可选）、教学楼（Building）、教室（Classroom）、图书馆与座位（Library/Seat）、课程（Course）、课程表/排课（CourseSchedule）、时间槽（TimeSlot）、预约（Reservation）、核销/扫码记录（Scan/Verification，可选）、评价（Review）、评价审核（ReviewAudit，可选/或字段承载）、报修工单（RepairTicket）、工单处理记录（RepairLog，可选）、协作小组（Team）、小组成员（TeamMember）、入组申请（TeamJoinRequest）、小组消息/聊天（TeamMessage/Chat）、学习计划（StudyPlan）、计划节点（PlanMilestone，可选）、资源条目（MarketItem）、交易意向/交易记录（Trade/Deal）、扫码设备（ScanDevice）、扫码授权（ScanDeviceAuth）。

3.5.2 E-R 关系说明（文字版，可配图）

（1）用户—预约：一名用户可创建多条预约记录（1:N）。  
（2）教室/座位—预约：一个空间资源在不同时间槽可对应多条预约（1:N）。  
（3）预约—时间槽：一条预约可包含多个连续时间槽（1:N 或 N:M，按实现选择）。  
（4）用户—评价：用户可对空间提交多条评价（1:N），公开展示受审核状态影响。  
（5）用户—报修工单：用户可提交多条报修（1:N），工单状态推进形成闭环。  
（6）协作小组—成员：小组与用户为多对多关系，可通过成员表维护（N:M）。  
（7）协作小组—学习计划：一个小组可对应多条计划（1:N）。  
（8）预约—学习计划：计划可关联预约，体现“计划-空间”联动（1:1 或 1:N，按实现）。  
（9）资源条目—交易记录：一个资源条目可产生多条交易意向/记录（1:N），交易双方与用户关联。

3.5.4 关键约束、索引与一致性设计（与真实表对齐）

（1）统一资源建模（教室/座位）  
`reservation`、`review`、`repair` 均采用 `resource_type + classroom_id/library_seat_id` 的模式统一建模，减少表数量并支持前端按“资源类型”统一展示。

（2）唯一约束防止脏数据  
例如 `classroom` 的 `uk_classroom_room (building_id,room_number)` 防止同楼同号重复；`library_seat` 的 `uk_library_seat (library_id,floor,row_num,col_num)` 保证座位坐标唯一；`team_member` 的 `uk_team_user` 防止重复入组。

（3）高频查询索引  
`reservation` 的 `idx_resource` 与 `idx_reservation_date` 支撑“按资源+日期查时段占用”；`review/repair` 的 `idx_resource` 支撑“按资源查评价/工单”；`team_message` 的 `idx_team_time_id` 支撑聊天分页。

（4）状态字段的枚举化与可追溯  
核心流程（预约、审核、工单）均采用 TINYINT 状态字段，并保留 `create_time/update_time` 与 `deleted` 逻辑删除字段，便于追溯与统计。

（5）一致性与幂等（设计要点）  
聊天表通过 `client_msg_id` + 唯一键实现幂等；预约核销链路通过记录 `qrcode_scan_time/device_uid` 等字段提升可追溯性。对于“创建预约—生成二维码—核销更新状态”等关键更新，应采用事务与必要的重复提交防护策略，避免状态错乱。
