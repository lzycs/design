第4章 系统详细设计与实现
本章在第3章总体设计的基础上，结合系统当前代码实现与真实数据库表结构，围绕用户端与管理端两条主线，对关键业务模块的设计思路、核心流程、接口实现与数据表映射进行说明。本文系统采用前后端分离架构，后端以 Spring Boot + MyBatis-Plus 为主，统一提供 REST API，并在小组聊天等场景引入 WebSocket 实时推送；数据存储以 MySQL 为主，Redis 用于缓存/临时状态支撑高频读取。
4.1 系统实现总体说明
本节从系统落地实现视角，说明平台在工程结构、业务分层与数据组织上的总体方案。为兼顾多业务并发、实时反馈与后台治理需求，系统在实现上遵循“统一接口规范、统一状态语义、统一资源抽象”的设计原则：接口层保证调用一致性，业务层保证规则可复用与可演进，数据层保证实体关系稳定与可追踪。通过该实现路径，系统能够在不牺牲可维护性的前提下支撑预约核销、反馈治理、协作通信、市场审核等多条业务链路并行运行。
4.1.1 分层架构与模块边界
后端采用 Controller-Service-Mapper 分层结构。Controller 层负责 HTTP 接口暴露、参数绑定与基础校验，并统一处理鉴权上下文与请求入口标准化；Service 层负责业务规则执行、状态流转控制与跨模块流程编排，是系统复杂逻辑的核心承载层；Mapper 层负责数据库读写、条件查询与必要联表操作，保证数据访问职责单一、查询语义清晰。该分层方式使预约规则校验、二维码核销、评价审核、工单处理、协作聊天等复杂流程能够集中在业务层实现，从而提升复用性、可测试性与后续迭代效率。
在调用边界上，系统遵循“Controller 不直接操作数据库、Mapper 不承载业务规则”的约束，避免逻辑下沉导致的耦合扩散。对于预约冲突检测、审核状态推进、消息已读回执等典型状态驱动场景，统一由 Service 层进行事务内编排与状态边界校验，确保并发条件下的数据一致性。异常处理方面，后端通过统一返回结构 `Result<T>` 对成功、业务失败与系统异常进行分层表达，前端仅需依据状态码与消息字段执行统一分支处理，降低了跨页面错误处理复杂度。
前端采用 Vue 3 + Pinia + Vue Router 组织页面、状态与路由。页面按业务域划分为预约中心、反馈评价、协作组队、学习计划、资源市场、校园导航、个人中心与后台管理入口，并通过路由守卫与角色信息联动控制可见性。状态管理层将用户会话、空间筛选条件、预约记录、协作消息等高频状态集中管理，减少组件间重复请求与状态漂移。请求通信层以 Axios 统一封装拦截器、鉴权头注入与错误兜底策略，使前后端在接口调用、权限校验与异常提示上形成一致交互体验。
4.1.2 核心数据建模特征
系统在“预约/评价/报修”等关键业务中采用统一资源模型，即通过 `resource_type` + `classroom_id/library_seat_id` 实现“教室/图书馆座位”两类资源的统一建模。其中，`resource_type` 用于区分资源类别，`classroom_id` 与 `library_seat_id` 用于承载具体资源主键引用，二者组合可在单套业务流程中完成多资源对象的统一识别与处理。
统一资源模型在业务表中的体现如下：
- 预约表：`reservation`，用于记录预约主体、目标资源、时间段与状态流转信息；
- 评价表：`review`（通用评价），以及 `classroom_feedback`（用于反馈页展示与审核流转）；
- 报修表：`repair`，用于记录故障描述、处理进度与完成结果。
该建模方式的直接收益在于减少表数量与重复字段设计，避免为不同资源类型维护两套独立流程。更重要的是，系统可在服务层沉淀统一的“可预约性判断、可评价性判断、可报修性判断”规则模板，提升规则复用能力；前端也可基于 `resource_type` 实现统一筛选、统一列表渲染与统一详情跳转，降低页面逻辑分叉。与此同时，统一模型仍保留了资源差异化扩展空间，当后续新增自习室、实验室等资源类型时，可通过扩展资源枚举与适配字段映射实现平滑接入，减少对既有业务链路的改动范围。
4.2 用户端核心模块设计与实现
4.2.1 用户注册与登录模块
用户端以统一账号体系为入口，完成注册、登录与身份识别。系统采用同一用户表承载学生、教师及管理类账号，通过角色字段区分权限边界。登录后，前端保存会话信息并加载个人中心相关数据，为后续预约、反馈、协作等功能提供身份上下文。
4.2.2 学习空间浏览与状态查询
该模块实现“楼宇-教室-时段”三级查询能力。用户可先浏览教学楼与教室基础信息，再结合课程安排与预约占用查看目标时段可用性。系统通过标准时间段模型统一前后端时间粒度，避免页面端拼装多源数据造成状态不一致。
对应数据主要来自 `building`、`classroom`、`course` 与 `time_slot` 表，其中 `classroom` 中的实时状态与运维状态字段可直接用于预约前风险提示。
4.2.3 预约与签到核销闭环
用户端核心业务围绕“预约创建-二维码生成-扫码核销-状态更新”展开。系统支持教室与图书馆座位统一预约，并在后端执行规则校验（周上限、单次时长）与冲突校验（课程/已预约占用）。
核销环节采用跨设备扫码模式：用户端生成二维码并轮询状态，设备端执行扫码核销，服务端完成有效性判定与预约状态流转。通过该机制将线上预约与线下到场确认连接起来，有效降低“预约未到场”带来的资源浪费。
该模块核心数据表为 `reservation` 与 `reservation_limit_config`，扫码授权相关数据由 `scan_device` 与 `scan_device_pair_token` 支撑。
4.2.4 反馈评价与报修服务

反馈模块覆盖“待评价生成-评价提交-审核展示”与“故障报修-进度跟踪”两类场景。系统在用户查询待评价列表时，基于已签到/已完成预约自动补齐待评价记录，减少漏评；评价提交后进入审核流，审核通过后在教室详情公开展示。
报修模块支持用户提交图文工单并持续跟踪处理状态。评价与报修共同构成使用后反馈通道，使平台从“可预约”进一步提升为“可治理、可改进”。
对应核心表为 `classroom_feedback`、`review` 与 `repair`。
4.2.5 协作组队、聊天与学习计划联动
系统将学习空间服务从个体场景扩展到团队协作场景。用户可在协作广场创建或加入小组，通过申请审核机制保证成员管理有序；在小组内可进行实时聊天、消息已读同步与文件共享。聊天模块引入 WebSocket 推送与消息幂等机制，兼顾实时性与一致性。
学习计划模块支持按用户、小组与预约维度查询，并可与预约记录直接关联，实现“目标-时间-空间”的协同组织。该联动设计提升了团队学习活动的执行效率与可追踪性。
相关数据由 `team_request`、`team_member`、`team_join_application`、`team_message`、`team_message_read` 与 `study_plan` 等表承载。
4.2.6 资源市场与校园导航扩展服务
为提升平台综合服务能力，用户端集成资源市场与校园导航两类扩展功能。资源市场支持资源发布、审核后上架、交易意向发起与线下面交确认，并结合校园信用分形成约束与激励机制。校园导航则依托楼宇与空间经纬度信息提供目标搜索、位置展示与路线规划，在路径服务不可用时可降级为直线导航，保障基础可用性。
该部分核心数据表包括 `resource_market_item`、`resource_market_trade`、`campus_credit`，以及导航相关的 `building`、`classroom`、`library` 地理字段。
4.3 管理端模块设计与实现
4.3.1 管理端登录、菜单与概览统计（RBAC）
管理端采用基于角色的访问控制机制。管理员登录后，系统按角色动态返回可见菜单，并在首页展示与当前权限相关的概览指标（如待处理工单、待审核评价、基础资源数量）。该设计保证了“最小权限可见”原则，避免非授权角色访问不必要的管理能力。
权限模型由 `admin_role`、`admin_permission`、`admin_role_permission` 与 `admin_menu` 表共同支撑，关键管理行为写入 `admin_operation_log`，为后续审计与责任追踪提供依据。
4.3.2 评价审核管理
评价审核模块以状态机方式推进：待审核记录经管理员处理后进入“审核通过”或“审核驳回”状态。系统在审核前进行状态边界校验，仅允许对待审核记录操作，从而避免重复审核或状态覆盖问题。
审核对象主要存储于 `classroom_feedback`，每次审核行为同步写入 `admin_operation_log`，保证审核过程可追溯、可复核。
4.3.3 报修工单管理与通知
报修模块围绕“受理-处理中-已修复/关闭”主流程进行管理。管理员可按状态和关键字筛选工单，并在处理时记录处理人、处理意见与处理时间。工单状态变化后，系统向报修人发送站内通知，形成用户可感知的处理反馈。
该模块核心数据为 `repair` 与 `notification`。此外，报修状态可与教室实时状态联动，避免用户在预约时选择维修中的空间，提升平台整体一致性。
4.3.4 资源市场审核
资源市场审核模块对用户发布内容执行准入控制，审核结论分为通过与驳回两类，并同步写入审核备注与操作日志。通过审核的资源进入可流转状态，驳回资源保留记录便于申诉与复核。
审核数据主要落在 `resource_market_item`，并通过 `admin_operation_log` 记录关键处理动作。权限策略与内容审核体系统一，便于后台治理模型简化。
4.3.5 基础数据管理与预约限制配置
后台同时提供基础资源维护与运行参数配置能力。前者覆盖教学楼、教室、课程等主数据，后者覆盖预约限制规则与扫码设备授权管理。通过将“静态资源管理”与“动态策略配置”统一纳入后台，系统可在不修改前端流程的前提下快速调整运营策略。
该部分主要依赖 `building`、`classroom`、`course`、`scan_device_pair_token` `reservation_limit_config`与`scan_device` 与等数据表。
4.4 本章小结
本章在第3章总体设计基础上，围绕平台已落地功能，对系统实现过程进行了模块化展开。面向用户端，重点说明了学习空间查询、预约与核销、评价与报修、协作组队与实时聊天、学习计划联动、资源市场与校园导航等业务的实现逻辑与数据支撑；面向管理端，重点说明了 RBAC 权限控制、评价审核、工单处理、资源审核与基础配置等治理能力的实现方式。通过双端协同设计，平台形成了“服务使用—反馈治理—运营管理”的完整业务闭环。

在工程实现层面，本章进一步验证了分层架构与统一接口规范的有效性：复杂规则集中于 Service 层处理，关键状态通过后端统一校验与流转，前端通过统一请求封装与状态管理保持交互一致性。在数据实现层面，统一资源模型与状态机机制使预约、评价、报修、审核等流程在同一数据语义下可追踪、可复核、可扩展，为后续优化与功能扩展提供了稳定基础。

综上，本章完成了从“总体方案”到“可运行系统”的落地说明，证明了平台在业务完整性、实现可行性与治理可控性方面达到预期目标，并为第5章从功能、性能与安全维度开展测试验证提供了实现依据与观测对象。
